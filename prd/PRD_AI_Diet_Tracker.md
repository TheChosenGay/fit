# AI 饮食记录助手 - 产品需求文档

## 一、产品概述

### 1.1 产品名称
食记AI（暂定）/ SnapCal

### 1.2 一句话定义
拍张照片即可自动识别食物并记录热量，AI帮你管理每日营养摄入的iOS App。

### 1.3 目标用户
- 25-40岁想减脂/控制饮食的白领
- 健身人群（需要精确控制碳水/蛋白质摄入）
- 轻度减肥需求的女性用户
- 有慢性病需控制饮食的人群（糖尿病、高血压等）

### 1.4 核心价值主张
- **对比通用AI工具的差异**：一拍即记，不需要打字描述，累积数据自动生成营养趋势报告——这是聊天工具做不到的
- **对比薄荷健康等竞品**：不需要手动搜索食物数据库，AI自动识别；无广告无社区，极简纯工具
- **核心体验**：从"每顿饭花2分钟手动搜索记录"变成"拍一张照片1秒搞定"

### 1.5 商业模式
- 免费版：每天3次拍照识别，基础热量记录
- 订阅版（¥25/月 或 ¥168/年）：无限次识别、详细营养素分析、AI饮食建议、周报/月报、数据导出
- 海外定价：$3.99/月 或 $29.99/年

## 三、MVP 功能详细设计（第一版）

### 3.1 模块A：拍照识别食物

**功能描述**：
- 用户拍摄或选择一张餐食照片
- AI自动识别照片中的所有食物品类
- 支持识别中餐、西餐、日料等主流菜系
- 一张照片可识别多个食物（如一份套餐内的多道菜）

**交互流程**：
1. 首页底部大按钮"记一餐" → 打开相机或选择相册
2. 拍照后显示加载动画（1-3秒）
3. 展示识别结果列表（食物名称 + 估算份量 + 热量）
4. 用户确认或修正 → 保存记录

**识别输出格式**：
```json
{
  "foods": [
    {
      "name": "番茄炒蛋",
      "portion": "一盘(约200g)",
      "calories": 180,
      "protein": 12,
      "carbs": 8,
      "fat": 11,
      "confidence": 0.92
    },
    {
      "name": "米饭",
      "portion": "一碗(约150g)",
      "calories": 195,
      "protein": 4,
      "carbs": 43,
      "fat": 0.5,
      "confidence": 0.95
    }
  ]
}
```

**技术方案**：
- 拍照：AVFoundation 自定义相机（简单模式即可）
- 食物识别：调用 Claude Vision API / GPT-4o（发送照片+prompt，返回结构化JSON）
- 备选：Google Cloud Vision API + 自定义后处理
- Prompt设计：要求返回食物名称、估算克重、热量、三大营养素

**为什么用大模型Vision而非传统食物识别API**：
- 传统API（如Clarifai）只能识别品类，无法估算份量和营养素
- 大模型可以理解盘子大小、餐具比例来估算份量
- 对中餐识别能力远超传统模型

**预计开发时间**：4-5天

---

### 3.2 模块B：热量与营养素估算

**功能描述**：
- AI返回的识别结果包含热量和营养素估算
- 用户可修改份量（±），系统自动换算热量
- 提供份量参考（一小碗/一大碗/一盘等常用单位）

**营养素展示**：
- 热量（kcal）- 核心展示
- 蛋白质（g）
- 碳水化合物（g）
- 脂肪（g）
- 膳食纤维（g）—— 完整版

**精度保障策略**：
- 内置食物营养数据库（中国食物成分表+USDA）作为校准参考
- AI识别后与数据库交叉验证，偏差>30%时用数据库值修正
- 用户反馈机制：标记"不准确"可改善后续识别

**技术方案**：
- 本地SQLite数据库（5000+常见食物营养数据）
- AI估算值与本地数据库加权融合
- 份量调整用滑块UI，实时计算营养素

**预计开发时间**：4-5天（含数据库建设）

---

### 3.3 模块C：每日饮食记录

**功能描述**：
- 按时间线展示今日所有已记录的餐食
- 分类：早餐/午餐/晚餐/加餐
- 每餐显示食物列表、照片缩略图、总热量
- 支持删除/编辑已记录的条目

**数据模型**：
```
MealRecord {
  id: UUID
  date: Date
  mealType: enum(breakfast, lunch, dinner, snack)
  photo: LocalFileURL
  foods: [FoodItem]
  totalCalories: Int
  totalProtein: Double
  totalCarbs: Double
  totalFat: Double
  createdAt: Date
}

FoodItem {
  id: UUID
  name: String
  portion: String
  weight_g: Double
  calories: Int
  protein: Double
  carbs: Double
  fat: Double
  isManuallyEdited: Bool
}
```

**技术方案**：
- Core Data 存储
- SwiftUI List展示
- 照片存储在App沙盒Documents目录
- iCloud同步（完整版）

**预计开发时间**：3-4天

### 3.4 模块D：每日营养总览

**功能描述**：
- 今日总热量 vs 目标热量（环形进度条）
- 三大营养素占比（蛋白质/碳水/脂肪）条形图
- 剩余可摄入热量显示
- 超标时红色警示

**UI设计要点**：
- 首页顶部卡片展示，一眼看清今日状态
- 环形进度条：绿色（<80%）→ 黄色（80-100%）→ 红色（>100%）
- 极简设计，数字要大、颜色要明确

**技术方案**：
- SwiftUI Canvas / Swift Charts 绘制图表
- 实时计算（每次记录变更时更新）

**预计开发时间**：2-3天

### 3.5 模块E：目标设定

**功能描述**：
- 初次使用时引导设定：身高、体重、年龄、性别、活动水平
- 自动计算基础代谢率（BMR）和每日推荐摄入（TDEE）
- 根据目标（减脂/维持/增肌）调整目标热量
- 用户可随时手动调整目标

**计算公式**：
```
BMR(男) = 10 × 体重(kg) + 6.25 × 身高(cm) - 5 × 年龄 - 5
BMR(女) = 10 × 体重(kg) + 6.25 × 身高(cm) - 5 × 年龄 - 161

TDEE = BMR × 活动系数
  久坐: 1.2
  轻度活动: 1.375
  中度活动: 1.55
  重度活动: 1.725

目标热量:
  减脂: TDEE × 0.8
  维持: TDEE
  增肌: TDEE × 1.1
```

**技术方案**：
- Onboarding引导页面（SwiftUI）
- UserDefaults存储基础设置
- 后续支持与Apple Health同步体重数据

**预计开发时间**：2天

- 编辑模式：点击已记录食物可修改份量

### 5.2 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|

| 小组件 | WidgetKit | 桌面/锁屏Widget |
| 最低版本 | iOS 16.0 | Swift Charts需求 |

### 5.3 数据源：食物营养数据库

**数据来源**：
1. 中国食物成分表（中国CDC发布，约2000+条目）
2. USDA FoodData Central（美国农业部，开放数据）
3. 常见外卖/连锁餐厅数据（手动整理Top 200）

**数据格式**：
```sql
CREATE TABLE foods (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  name_en TEXT,
  category TEXT,
  calories_per_100g REAL,
  protein_per_100g REAL,
  carbs_per_100g REAL,
  fat_per_100g REAL,
  fiber_per_100g REAL,
  common_portion TEXT,
  common_portion_g REAL,
  source TEXT
);
```

**初期数据量目标**：5000+条目覆盖90%日常饮食场景

### 5.4 关键技术风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 食物识别不准确 | 用户体验差，数据不可信 | 允许手动修正+用户反馈改善prompt |
| 中餐识别难（混合菜品） | 热量估算误差大 | prompt中强调中餐场景，提供份量参考 |
| API调用成本（每次含图片） | 图片Token消耗大 | 压缩图片至720p再上传，缓存常见食物结果 |
| 热量估算精度 | 用户质疑可信度 | UI标注"估算值"，提供±范围，允许修正 |
| 食物数据库维护 | 数据过时 | 使用权威数据源，定期更新 |
